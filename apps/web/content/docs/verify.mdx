title: Verify
description: Check ClickHouse queries and migrations against the real cluster, from the CLI or your agent
---

`gozzle verify` is a read-only test harness for ClickHouse changes. Point it at a
SQL file (or your branch's changes) and it routes a `SELECT` to query diagnosis
and an `ALTER` to a migration dry-run against the real cluster, then **exits
non-zero on findings** so it drops into any pre-commit hook or CI step.

It uses the same read-only connection as the rest of gozzle
([read-only enforcement](/docs/read-only)): set `GOZZLE_CLICKHOUSE_URL` and
credentials in the environment.

```bash
gozzle verify ./queries/revenue.sql        # one file
gozzle verify --changed                     # files changed on this branch
gozzle verify --diff origin/main...HEAD     # files in a git range (use in CI)
gozzle verify --all                         # the whole configured SQL surface
gozzle verify ./q.sql --strict              # also fail on advisory findings
gozzle verify ./q.sql --json                # machine-readable output
```

- A file containing a `SELECT`/`WITH … SELECT` is diagnosed with
  [`diagnose_query`](/docs/diagnose-query); an `ALTER TABLE` is assessed with
  [`dry_run_migration`](/docs/dry-run-migration). The original query/migration is
  never executed.
- `--changed` resolves uncommitted and untracked changes; `--diff <range>` uses a
  git range; `--all` walks the [`gozzle.yaml`](#gozzleyaml) globs.

### Exit codes

| Code | Meaning |
| --- | --- |
| `0` | Clean, no failing findings (advisories alone pass without `--strict`). |
| `1` | Findings that fail the gate: a **proven** query problem, or a migration that rewrites parts. |
| `2` | Operational error, bad file, unknown statement, or no connection. |

Because the gate is just an exit code, you can wire it into your own CI:

```bash
gozzle verify --diff origin/main...HEAD
```

## gozzle.yaml

`gozzle verify` works on individual files with no configuration. `gozzle.yaml` is
an optional, versioned project file that unlocks the two things a single file
can't express: **which files make up your SQL surface**, and **what your tables
are supposed to guarantee**. It's local config committed to your repo, not a
hosted service. gozzle walks up from the working directory to find it.

```yaml
# gozzle.yaml
database: default
queries:
  - app/**/*.sql
  - dashboards/**/*.sql
migrations:
  - migrations/**/*.sql
assumptions:
  events:
    unique_by: [event_id]
```

Three things it controls:

- **`database`**: the default database for unqualified table names. (A
  `GOZZLE_CLICKHOUSE_DATABASE` in the connection takes precedence.)
- **`queries` / `migrations`**: globs that define your SQL surface. They drive
  `gozzle verify --all` and filter `--changed` / `--diff`, so gozzle only
  verifies files you've marked as queries or migrations.
- **`assumptions.<table>.unique_by`**: declares the columns a table is *supposed*
  to be unique by. This is what powers the [read-path proof](#the-read-path-proof)
  below: gozzle checks whether queries actually honour that uniqueness against
  live data.

Why create it:

- **Run `verify` over a defined SQL surface.** `gozzle verify --all` and
  `--changed` use the globs instead of you naming files by hand, ideal in CI.
- **Catch silent overcounting.** The read-path proof flags queries that trust a
  uniqueness your data doesn't currently hold.

Without it, `verify` still runs query diagnosis and migration dry-runs per
file, you just don't get the `--all` / `--changed` surface or the read-path
proof.

## The read-path proof

Duplicates inside a `ReplacingMergeTree` are expected storage behaviour. The
bug is a query that reads such a table **as if rows were already unique**. When a
query reads a table you've declared `unique_by` *without* `FINAL`, gozzle checks
whether the current data actually violates that assumption:

```text
Read-path proof:
- [violated] analytics.events is read without FINAL and trusted as unique by
  (event_id), but currently has 2,400,000 duplicate row(s) by sorting key.
  This query can overcount.
```

A violation fails the gate. Large tables that can't be confirmed within the scan
guard are reported as `[unknown]` rather than blocking.

## Prove a rewrite is equivalent

When you (or an agent) rewrite a query, `gozzle equivalent` proves whether the
two return the **same result**: same multiset of rows, order-independent,
computed entirely in ClickHouse, no data copied.

```bash
gozzle equivalent ./old.sql ./new.sql      # exit 0 equal · 1 different · 2 indeterminate
```

It compares result shapes first, then diffs the rows with `EXCEPT ALL`, and on a
difference returns a capped sample of the diverging rows. Non-deterministic
queries (`rand`, `now`, …) and comparisons too large for the scan guard return
`indeterminate` rather than a misleading "equal". Also available as the
`verify_equivalent` MCP tool for agents.

## Make your agent verify automatically

Two ways, increasing reliability:

**Skill**: `gozzle skill` prints an instruction telling the agent to verify with
gozzle before answering. Portable across hosts, but discretionary (agents can
forget).

```bash
gozzle skill            # instruction + where it goes, for every host
gozzle skill cursor     # one host: claude, cursor, or codex
```

It places into CLAUDE.md (Claude Code), `.cursor/rules/gozzle.mdc` (Cursor), or
AGENTS.md (Codex), and tells the agent to treat gozzle's verdict as ground truth.

**Hook (deterministic)**: `gozzle hook` prints a Claude Code PostToolUse hook
that runs after *every* file edit, so a changed `.sql` file is verified
automatically, no reliance on the agent choosing to. On findings it blocks and
shows the verdict to the agent; non-SQL edits and setup issues are ignored.

```bash
gozzle hook             # the settings.json recipe (--local for npx)
```
