This walks you from zero to your first **verdict with proof** — gozzle finding (or
clearing) duplicate rows in a real ClickHouse table. It takes a few minutes.

## Prerequisites

- **Node.js 22+**
- A **ClickHouse** you can reach, with a user that can run `SELECT` (a read-only
  user is ideal — gozzle never needs more)
- An MCP host: **Claude Code**, **Cursor**, or **Codex**

<Steps>

<Step>

### Install

```bash
npm install -g @gozzle/cli
```

This installs two binaries: `gozzle` (the CLI) and `gozzle-mcp` (the MCP stdio
server your AI host talks to).

</Step>

<Step>

### Configure your AI host

Print the MCP config block for your host and the file it belongs in:

```bash
gozzle init           # all hosts
gozzle init claude    # or: cursor, codex
```

Paste the printed block into the named file, then replace the connection values
with your ClickHouse details. Use a **read-only** user — gozzle forces
[`readonly=2`](/docs/read-only) on every query regardless.

```json
{
  "mcpServers": {
    "gozzle": {
      "command": "gozzle-mcp",
      "env": {
        "GOZZLE_CLICKHOUSE_URL": "https://your-cluster.clickhouse.cloud:8443",
        "GOZZLE_CLICKHOUSE_USER": "gozzle_readonly",
        "GOZZLE_CLICKHOUSE_PASSWORD": "your-password",
        "GOZZLE_CLICKHOUSE_DATABASE": "default"
      }
    }
  }
}
```

Restart your AI host so it picks up the new server.

</Step>

<Step>

### Connect

Ask your agent to run the `connect` tool, or just say:

> Use gozzle to connect to my ClickHouse.

You should see the connection confirmed, with read-only enforcement reported:

```text
Connected read-only check complete.
No data leaves this machine.

Read-only enforced by gozzle: yes (readonly=2)
```

</Step>

<Step>

### Inspect a table

Point gozzle at a table to see its physical layout and which checks it is
eligible for:

> Inspect the table `analytics.events` with gozzle.

`inspect_table` reports the engine, sorting/partition keys, parts, and row
counts — and tells you whether the table can be checked with
[`verify_dedup`](/docs/verify-dedup).

</Step>

<Step>

### Verify a change — `gozzle verify`

`gozzle verify` is the everyday verb and gozzle's leading feature: point it at a
changed `.sql` file or your branch's diff and it proves whether the change is
safe against the real cluster, **exiting non-zero on findings** — so it drops
straight into a pre-commit hook or CI.

```bash
gozzle verify ./queries/revenue.sql        # one file
gozzle verify --changed                     # everything changed on this branch
```

A `SELECT` is diagnosed with `EXPLAIN`; an `ALTER` is dry-run — the original is
never executed. Add a [`gozzle.yaml`](/docs/verify#gozzleyaml) with your tables'
`unique_by` assumptions to unlock the read-path proof, which catches a query
that reads a deduplicated table without `FINAL` and can silently overcount:

```text
Read-path proof:
- [violated] analytics.events is read without FINAL and trusted as unique by
  (event_id), but currently has 2,400,000 duplicate row(s) by sorting key —
  this query can overcount.
```

That is the proof moment: a claim a plain `SELECT count()` can't make, backed by
ClickHouse's own semantics. Run `gozzle skill` so your agent verifies changes
automatically.

</Step>

</Steps>
