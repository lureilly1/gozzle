---
title: verify_dedup
description: Prove whether a ReplacingMergeTree table currently holds duplicate rows
---

`ReplacingMergeTree` (and its `Shared`/`Replicated` variants) deduplicates rows
with the same sorting key **lazily** — during background merges, per partition.
Until a merge runs, duplicates are still physically present, so a plain
`SELECT count()` is misleading and `SELECT ... FINAL` hides the problem.

`verify_dedup` returns a **verdict plus evidence**: how many rows a background
merge versus `SELECT ... FINAL` would collapse, and a sample of the
worst-offending keys.

## Usage

Ask your AI client:

> Use Gozzle to verify deduplication on `analytics.events`.

Parameters:

- `table` — `table` or `database.table`.
- `sampleLimit` — how many duplicated keys to return as evidence (default `5`, max `50`).

## Example output

```text
Table: analytics.events
Engine: SharedReplacingMergeTree
Dedup key (ORDER BY): id
Partitioned: no
Total rows: 7

Verdict: 3 duplicate row(s) across 2 sorting-key group(s) would be collapsed by a merge or FINAL.
Most-duplicated key has 3 copies.

Sample duplicated keys:
- [partition all] id=1 -> 3 copies
- [partition all] id=2 -> 2 copies
```

## Merge vs FINAL on partitioned tables

Two operations remove duplicates, and on a partitioned table they remove
**different** numbers of rows — so Gozzle reports both:

- **Background merges** run *per partition*. Duplicates sharing a sorting key
  within one partition are collapsed; identical keys in different partitions are
  **never** merged, so they are permanent. This is the eventual physical floor.
- **`SELECT ... FINAL`** deduplicates *globally* by sorting key by default
  (`do_not_merge_across_partitions_select_final = 0`), so it can collapse more
  rows than merges ever will.

```text
Verdict: duplicates differ by scope on this partitioned table:
- Background merges collapse 1 row(s) across 1 per-partition group(s) (the eventual physical floor).
- SELECT ... FINAL collapses 3 row(s) (it deduplicates globally by sorting key).
...
Notes:
- 2 of the FINAL-collapsible rows are cross-partition duplicates that background merges never remove.
```

<Callout title="Why this matters">
A naive `count()` over-counts duplicates; a single per-partition number
under-counts what a user sees from `FINAL`. Gozzle reports both, so you know the
storage floor and the query-time result. For a non-partitioned table the two are
equal and the verdict collapses to one line.
</Callout>

## Eligibility

`verify_dedup` runs only when `inspect_table` reports it as eligible:

- the engine is a `ReplacingMergeTree` family engine, and
- the table is not `Distributed` (run it against the local table on each shard), and
- the table has a sorting key.

For ineligible tables, Gozzle returns a clear reason instead of a misleading
count.
